# SRUI Architecture

# 1. Overview

SRUI is a screen-reader-first UI toolkit for .NET. It maintains a retained semantic tree of widgets, processes keyboard input against that tree, and emits structured accessibility events describing what the user should perceive. It produces no pixels and never will: visual rendering is permanently out of scope. What an application does with the events — synthesize speech, drive a braille display, back a platform accessibility API — is decided by the readers attached to it (section 7).

The toolkit is pure C# from the application surface down to the engine. Native code appears only at the platform edges — SDL3 for the window and keyboard, Prism for speech, cosmos/phonon for game audio — each reached through hand-written P/Invoke (section 11).

# 2. Design Principles

## 2.1. Sans-IO engine

The engine owns no platform resources: no TTS engine, no clipboard, no window, no event loop, no file access, no wall clock (time is injected). The host pushes input in and drains events out. Platform services the engine needs (clipboard) are injected behind interfaces. This keeps the engine headless and deterministic under test — a headless `SruiApp` is a complete, drivable application with no native dependencies.

## 2.2. One object per widget

A widget is a single object: the application-facing surface, the widget's state, and its input behavior live together in a `Widget` subclass, and that object is its node in the semantic tree. Identity on the public surface is the object reference — there are no handles to go stale, and events reach their widget directly. The node's label (the golden six, section 4.2) needs no synchronization: its derived fields are functions of widget state, pulled from the widget whenever an announcement needs them; widget state (an edit box's text, a list's selection) is read and mutated through the widget's own properties, and user-driven changes are reported through its events. Application state synchronization is the program's responsibility (a future reactive layer may automate it).

## 2.3. Structured events out; readers render

The engine never speaks. It emits structured accessibility events carrying complete semantic payloads (`AccessibilityEvent`, carrying widget references and label snapshots). Speech composition is provided as a pure library (`SpeechRenderer`: event in, utterance string out) that readers may use or bypass. This is what lets one event stream feed a self-voicing reader, a braille reader, a UIA provider, and a test harness simultaneously.

## 2.4. Policy-free

Application vocabulary stays out of the engine: no application command names, no command categories, no persistence formats, no application-global shortcuts. The engine supplies mechanisms (key combos, binding maps, conflict detection) and the host supplies policy. Serialization of user configuration is host-owned; the engine exposes stable string forms (for example `"ctrl+shift+s"` for key combos) and plain data structures.

# 3. Layering

| Layer | Where | Role |
|---|---|---|
| Engine | `Srui.Net`, namespace `Srui.Core` (internal) | Retained tree, focus/navigation, input dispatch, the output event queue and coalescing, the text engine (rope, grapheme/word/line navigation, editor). Pure; no platform dependencies; no widget behavior of its own. |
| Widget layer | `Srui.Net`, namespace `Srui` | The public API: `SruiApp` (engine ownership, readers, dialogs, the loop), the `Widget` base class and the built-in widgets (each carrying its own behavior), `Dialog` and the canned dialogs, `Ticker`, `MultiAppHost` (several apps in one window, section 10), the input and event vocabulary (`InputEvent`, `AccessibilityEvent`, `Key`/`KeyCombo`). |
| Host | `Srui.Net` (`SdlHost`) | SDL3 window and event pump: a focus-receiving window, physical→logical input translation, the physical key stream (section 6.4), and the system clipboard. Optional — a headless `SruiApp` accepts logical inputs from any host. |
| Speech | `Srui.Net` (`Speech`, `SpeechReader`) | Speech output through Prism (vendored under `native/prism`), which routes to the running screen reader or platform TTS. `SpeechReader` is the reference reader; a windowed `SruiApp` installs one out of the box. |
| Game audio | `Srui.Audio` over cosmos.dll | 3D positioning (Horizon-style pan/volume or Steam Audio HRTF), ordered effect chains on sounds and groups (`SetFxChain`: a `SoundEffect` list with repeatable kinds — order-based and multimode filters, HRTF binauralizer/positional, disperser, convolution reverb, delay, distortion, vocoder, EQ), pitch tweens, offline time-stretch. All DSP stays native (cosmos.dll: miniaudio + vendored cosmos nodes, decoding wav, flac, mp3, vorbis, and opus); C# orchestrates with zero steady-state allocation. Everything is edge-triggered except time-based automation, which `SoundManager.Tick` applies: `SruiApp` owns an on-demand manager (`SruiApp.Audio`) and ticks it from the event loop at the loop cadence (`SruiApp.LoopWaitMs`, default 2 ms at idle); loop-less consumers tick manually. The device period is requested at manager creation — `SoundManager(periodFrames)`; the default 128 frames (~2.7 ms at 48 kHz) favors trigger-to-ear latency, heavy scenes request more (e.g. 512) for mixing headroom. The period is fixed for a device's lifetime — WASAPI sets it at client initialization and the Steam Audio convolvers are sized to the granted block — so `SoundManager.Reconfigure(periodFrames)` (surfaced as `SruiApp.AudioPeriodFrames`: read the granted period, write to request a new one) rebuilds the engine in place: every sound and group keeps its object and state — source, position, effect chain, transport, playback cursor, running tweens — replayed against the new device, and the decode cache survives the swap, so nothing re-decodes. Output gaps briefly while the device restarts; change the period at silence-tolerant moments. `SruiApp.AudioSampleRate` reads the device rate. Measured scale (AudioBench, Intel i7-12700F, 48 kHz): the 128-frame period carries roughly 400 sources across 200 HRTF positions; 2000 sources across 200 positions need 512 frames. |
| Readers | application-side | Consumers of the accessibility event stream (`IReader`): self-voicing speech, braille, UIA provider, logging/test readers. |

`SruiDemo` wires the whole stack together end to end: SDL events in, Prism speech out; `SruiTasks` drives the same stack through the extension paths of section 5, with built-ins subclassed and panels built as composite `Group` subclasses; `MultiAppDemo` runs three apps in one window under a `MultiAppHost` (section 10). A reactive layer over the retained API is anticipated but deferred.

# 4. The Semantic Tree

## 4.1. Nodes and widgets

The tree owns all nodes in a dictionary keyed by an engine-internal handle allocated from a monotonic counter and never reused; resolution is O(1) and a stale handle resolves to nothing rather than to a recycled node. Each node stores its parent, an ordered child list, its semantic label, and a reference to the `Widget` object that embodies it — the object that handles the node's input and receives its events. Handles never cross the public surface: applications hold widget objects, and an event queued for a widget that has since been removed resolves to nothing and is dropped at dispatch.

## 4.2. The golden six

Every node's label carries the six semantic properties a screen reader announces, in NVDA order: name, role, value, states, description, shortcut. The role is carried as its spoken text ("button", "edit read only multi line", or a custom widget's own word like "table"); an empty role text announces role-less — the usual choice for game surfaces whose identity is their name. Which keys a widget consumes during normal interaction — the basis of shortcut conflict detection — is the widget's own declaration (`Widget.ReservesKey`), not a property of the role text. `States` is a bitflag set of conditions the user cannot directly change (disabled, required, warning, hidden, with-help), each settable as a widget property; with-help is derived from `Widget.KeyHelp` being set (section 5) rather than toggled directly. The shortcut slot holds the widget's shortcut list (section 8); announcements speak the first shortcut added, in its spoken display form. A node's name is optional; a nameless node announces as role and value only, for widgets whose identity is carried by their container. Reader-facing events carry the label as an immutable snapshot (`WidgetInfo`) taken at emission time, so readers never race the tree. The six split by ownership: name, role, description, the flag states, and shortcuts are authored — set through widget properties and stored on the node — while value and the dynamic state text (spoken before the flag states: "no filter", "3 of 7") are derived — functions of widget state (`Widget.ValueText`/`Widget.StateText`, cheap and side-effect-free), never stored, and pulled from the widget at the moment a snapshot is composed. A derived field therefore cannot go stale: a list item's line composed from mutated application state reads correctly with no synchronization call anywhere.

## 4.3. Mutation

Programs build UI by constructing widgets (each widget inserts its node under its container at construction) and change it through widget properties. Mutations are synchronous and immediately visible. Mutations that affect what the user is perceiving cause appropriate accessibility events; mutations elsewhere in the tree are silent. A programmatic change to a focused widget's single value state — a slider's value, a list's selection, the active tab — speaks exactly what the equivalent user-driven change speaks (the new value alone, so a ticking progress slider stays terse), because the property setter and the input handler share the emission. Every other label property changed on the focused widget speaks as a delta too: the new name, role text, value, or description travels in a `LabelChange` event, and a state flag's transition in a `StateChange` event ("unavailable", "not required" — the same vocabulary focus announcements use), so non-speech readers receive the payload rather than a rendering. A property change never causes a full announcement — that is reserved for focus arriving on a widget (moves, recovery, and explicit re-announcements). Dynamic state text, shortcut changes, and hiding are structural and stay silent: state text and shortcuts ride the next focus announcement, and hiding the focused widget reads as the focus recovery it causes. Disabling the focused widget moves nothing: it speaks "unavailable" and focus stays, because disabled widgets remain reachable (section 8). Removing a widget removes its subtree, and focus recovers to the nearest surviving focusable node with an announcement.

## 4.4. Layers

The tree maintains a stack of layers for modal UI. Each layer has its own roots, focus, and default primary/cancel widgets (Enter and Escape targets). Only the top layer is navigable. Pushing a layer opens a dialog or palette; popping it removes the layer's nodes and restores the previous layer's focus. The base layer cannot be popped.

Dialog conventions live in the widget layer, not the engine: there is no dialog role (the prompt is a Label preceding the widgets, announced as a context label via the focused-with-context re-announcement), Escape closes a dialog automatically when no cancel widget claims it, and the canned dialogs (message, confirm, custom button rows, reviewable read-only status text) are `SruiDialogs` helpers. Canned dialog buttons carry the Windows-conventional Alt+letter activation shortcut, assigned first-free-letter within the dialog and spoken with the button; widget shortcuts only match within the active layer, so these never collide with the window underneath.

# 5. Widget Behavior and Authoring

Widget behavior is virtually dispatched on the widget object itself: when input reaches a focused widget, the engine calls its `OnInput` override, which mutates the widget's own state (the label follows by itself — its derived fields are functions of that state) and emits events. The built-in widgets (button, checkbox, single- and multi-line edit box with read-only, listbox, filter listbox, slider, tab control, shortcut field) are ordinary subclasses of the public `Widget` base and double as the reference examples for the three extension paths:

- **Composition** — subscribe to a widget's events (`Changed`, `Activated`, `Toggled`…).
- **Subclassing a built-in** — override its `On*` methods, keeping the base call so composition subscribers still fire (the demo's sound-augmented lists); withholding it is itself the tool when a subclass gates the event, as a press-twice-to-confirm button does. A subclass input handler that reshapes base-owned state reaches first for the item operations below (a list's items) or the protected silent counterparts of the public setters (`ListBox.SetItemsSilently`, `EditBox.SetTextSilently`) and owns the editorial announcement — mutate silently, then emit; the public setters are for programmatic changes and speak the changed property when focused (section 4.3).
- **Authoring a new widget kind** — subclass `Widget` directly: pass the spoken role text to the base constructor, override `OnInput` to claim input, expose the label's derived fields as functions of widget state (override `ValueText`/`StateText`), describe the perceptual result with `Announce`/`AnnounceItem` (or `Promulgate` for a raw structured event), and defer program notifications with `Post`/`PostChanged` so handlers never run inside input dispatch. Override `ReservesKey` to declare the combos the widget consumes, and the widget participates in bind-dialog conflict warnings like any built-in. An authored widget is a full citizen of the tab ring, focus recovery, dialog layers, shortcuts, and every attached reader (the demo's table widget is the worked example).

A `CustomWidget` is the degenerate case: focusable, role-less, no behavior — its interaction lives entirely in host bindings over the physical key stream (section 6.4).

Nonstandard keys and actions — anything a user could not predict from the widget's name and role — are documented through `Widget.KeyHelp`, a text property on every widget. Setting it adds the with-help state ("with help" in announcements, a `StateChange` transition when toggled on the focused widget) and claims plain F1 while the widget is focused: the framework shows the text in the canned reviewable status dialog, deferred to drain time like any dialog opening. The claim sits between the widget's own `OnInput` (which wins, so a widget kind that uses F1 itself keeps it) and framework navigation; the `Widget.ReservesKey` base implementation reports F1 reserved while help is set, which is why overrides keep the base call. Key help is the sanctioned home for what would otherwise be smuggled into the description and re-spoken on every focus visit (docs/accessibility-guidelines.md, section 8).

The list widgets are generic over their items (`ListBox<T>`, `FilterListBox<T>`; `T` a class implementing `IListItem`), so state-bearing subclasses read their items back typed, without casts; the non-generic `ListBox` and `FilterListBox` are the plain-`IListItem` forms and carry the string convenience overloads, which wrap plain strings into items. A list does not claim Enter — it falls through to the layer's primary widget, the Windows dialog convention; a `ListBox` whose items are themselves the things to open (a file manager, a level select) opts into the Explorer convention with `activateItems: true`, claiming Enter and raising `Activated` for the selected item. A multi-select list (`multiSelect: true`) announces as "multi select list" and lets the user check items independently of the selection: navigation speaks "checked" after a checked item's text and nothing after an unchecked one (the absence is the signal), and toggling speaks the new state alone with the same words a check box uses. Enter toggles by default — claiming it away from the layer's primary, which is why `multiSelect` cannot combine with `activateItems` in that mode — and `toggleWithSpace: true` moves the toggle to Space instead, freeing Enter (and re-admitting `activateItems`) at the cost of multi-word typeahead, since Space no longer feeds the typeahead buffer. Checked state lives with the item object: the structural operations drop a removed or replaced item's check, and replacing the item list keeps checks on item objects that survive into the new list. The programmatic surface is `IsChecked`/`SetChecked` (the setter speaks like a user toggle only when the item is the focused selection, and raises no event — the program already knows), `CheckedItems` in list order, and the `ItemToggled` event for user-driven toggles. A subclass refuses a user toggle per item by overriding `CanToggle`: returning false leaves the state, the announcement, and `ItemToggled` unfired — the framework says nothing, so the override should announce its reason rather than leave a dead key. `SetChecked` is the program's and is never refused. An item exposes its line (`Text` — what readers speak, typeahead matches, and filtering searches; read live wherever the framework needs it, so it must stay cheap and side-effect-free) and scores itself against a filter query (`FilterScore` — null excludes, higher sorts first, ties keep ordinal text order; the default is the built-in fuzzy match, and an empty query bypasses scoring entirely). Application item types implement the interface directly, composing their line from their own state — mutating that state needs no sync call, because the line is pulled fresh at announcement time (section 4.2). The item operations split announcement ownership: the framework owns the structural consequence, the caller owns the editorial. `RemoveAt` of the focused selection speaks the survivor the selection lands on exactly as an arrow move would — or "empty" when the last item went, the same word an empty list answers arrow keys with — while a caller wanting "Deleted X." announces it before the call; removing an unselected item is silent, as are `Insert`/`Add` (the selection keeps its item — except into an empty focused list, where the selection lands on the new item and speaks it) and `SetItem` (replacing one item object with another; the caller speaks the delta, if any).

A tab control owns its panels' visibility once they are attached (`TabControl.AttachPanels`, one panel per tab): the active tab's panel shows, the rest hide and leave navigation, re-synced on every switch — at drain time for user-driven switches (panels are other widgets, so input dispatch never touches them) and synchronously for programmatic `ActiveIndex` changes. Panels are created after the tab control, so they sit below it in tree order.

The text engine under the edit box deserves note. Text lives in a rope (a balanced tree of string leaves, O(log n) local edits); positions are UTF-16 code units, exposed on the editing surface (`CursorPosition`, `Selection`) with setters that clamp to the text and snap backward onto grapheme cluster boundaries, so a position can never land inside a surrogate pair, a combining sequence, a CRLF, or an emoji ZWJ sequence. Cursor movement is grapheme-aware (text-element segmentation, so combining sequences, surrogate pairs, and CRLF move as single characters), and word navigation uses character-class scans in three flavors: code-editor boundaries for speech context, Windows-style word starts for Ctrl+arrows, and Notepad-style word extents for Ctrl+Shift selection and word deletion. The edit box exposes the engine's geometry as pure position queries — word starts and extents, the word at a position, line bounds and line text, and 0-based line/column ↔ offset conversion (`LineCount`, `LineColumnAt`, `PositionAt`; a column is the code-unit offset from its line start) — all clamping and grapheme-snapping like the position setters, and none of them announcing anything. Movement, by contrast, is announced: the `Move*` family (`MoveLeft` through `MoveToDocEnd`) runs the same handling path as the key each method names, so a programmatic word jump speaks the word Ctrl+Right would, an edge hit reports the boundary, and a collapsing selection behaves like an arrow key — when the widget is focused; unfocused, the cursor moves silently. Programmatic editing has two entry points, both ignoring read-only (which guards user input, not the program, like the `Text` setter): `InsertText` is the announced form — insert at the cursor, replacing an active selection, spoken like typing the text — and `ReplaceRange` is the silent structural splice under find-and-replace and its kin, whose caller owns the announcement (the `SetItem` convention) and which maps the cursor and selection endpoints through the edit: positions before the range keep their place, positions after it shift with the length change, positions inside it land at the end of the new text.

# 6. Input

## 6.1. Vocabulary

A key combo (`KeyCombo`) is a physical key plus ctrl/alt/shift modifier state, with a spoken display form ("control shift s") and a compact config form ("ctrl+shift+s") that round-trips through parsing (`KeyCombo.TryParseConfig`). The string forms exist for serialization and speech only: every API that accepts a combo (shortcuts, key bindings) takes a `KeyCombo` value, so a malformed combo string surfaces where configuration is parsed, not as a runtime failure at bind time. A logical input (`InputEvent`, an `InputKind` plus payload) is the semantic layer above it: navigation (next/previous, hierarchy moves, mnemonic), widget verbs (activate, move by character/word/line, select, type, delete, clipboard), `SpeakFocus`, `Dismiss`, and `RawKey` for combos with no semantic mapping. The default physical-to-logical map lives in the SDL host and is host-replaceable; the host is responsible for translating platform key events into logical inputs.

The mapping is many-to-one (shift+backspace and ctrl+backspace both mean word-delete), so every logical input from a real keyboard also carries its physical provenance: the combo that actually produced it travels on the event, including for typed characters, whose suppressed keydown is paired with the following text-input event (shift+q is representable even though the typed rune is just "Q"). Shortcut matching and shortcut-field capture read the provenance directly and never reverse-map; a canonical reverse map (`KeyCombo.FromInput`) serves only synthetically injected inputs, which carry none. Widget code matches by the same rule: `InputEvent.Is` answers whether an input was produced by a given combo (or bare key), provenance first with the canonical fallback, so an input handler claims shift+up as shift+up rather than by the logical kind it happens to map to; `InputEvent.IsChar` compares a typed character by full codepoint, so an astral character can never alias a BMP one.

## 6.2. Dispatch

Input events are pushed to the app one at a time (`SruiApp.HandleInput`); there is no frame granularity. Each event is dispatched in a fixed claim order: the focused widget's `OnInput` gets first claim, then framework navigation and layer defaults (tab ring, hierarchy navigation, primary/cancel), then widget shortcuts (section 8), then dialog dismissal, then the host's own bindings (`UnhandledInput`). Whatever handles the event emits the corresponding output events; queued output is delivered when the app drains (`DispatchEvents` — the event loop does this every iteration).

## 6.3. Time

The engine has no clock; the host feeds one through `SetNow` (monotonic milliseconds) — the event loop does this every iteration, and headless hosts do it themselves. Typeahead timeouts and tickers are both observed there.

One event-loop iteration is public as `SruiApp.Tick`: advance the engine clock from the app's own stopwatch, advance audio automation, pump and dispatch window input (absent on a headless app, which returns without waiting), drain queued events to the readers — `Run` is exactly `Tick` in a loop, and `Quit` makes `Tick` return false. A host embedding several apps ticks each of them from its own loop: at most one app owns a window (SDL's event queue is process-global), and the rest run headless, taking input the host forwards through `HandleInput`/`HandleKey`. A tick-driven app runs on real time; deterministic hosts (tests) drive `SetNow`/`DispatchEvents` directly instead, since `Tick` would overwrite an injected clock. Applications read the same clock back through `SruiApp.Now` (the engine clock: frame-coherent, at loop resolution) and `SruiApp.PreciseNow` (a live sub-microsecond reading of the clock behind it; on headless apps it follows the engine clock so injected time stays deterministic). A ticker (`SruiApp.StartTicker`) raises its `Tick` event each time its interval elapses, at `SetNow` resolution, drift-tolerant: a late check fires once rather than bursting to catch up.

## 6.4. The physical key stream

Alongside the logical stream, the host layer reports raw key transitions for game-style input: the initial press, auto-repeats, and the release of every key with a key-combo form (bare modifier taps have none). This stream never enters the engine — the engine is semantic, and key bindings are host policy (section 9) — and it is independent of the claim order: a press is reported even when the corresponding logical input is consumed. Because a release can never be consumed and a press can never be hidden, the pairing a game relies on is unconditional. The host also signals the window losing keyboard focus, which orphans held keys: game state that tracks them must be zeroed, as the releases will never arrive.

The widget layer builds per-widget key bindings on this stream: handlers attach to any widget for a combo and phase (press, repeat, release) and fire while that widget is focused (`Widget.BindKey`), with unclaimed transitions falling to an app-level hook (`UnhandledKey`). Press and repeat match the exact combo; release matches the key alone, so a modifier pressed mid-hold cannot orphan the release.

# 7. Output Events and Readers

## 7.1. Event streams

The engine produces a single ordered stream of output events. Accessibility events (`AccessibilityEvent`, public) describe perception: focus moved (with the widget, a golden-six snapshot, any context labels, and the cause — `FocusCause`: user navigation, a shortcut jump, programmatic focus, recovery, layer restore, or a re-announcement — so a reader can treat focus the user moved differently from focus that came to the user), text typed or deleted, cursor moved, selection changed, list item changed (with position), tab changed, slider changed, check box toggled, filter results changed, boundary hit, an editor input with nothing to act on (empty-editor navigation, a delete with nothing beside the cursor, a selection pinned at a text edge), label property changed and state flag changed (the focused-widget deltas of section 4.3), plus a free-form announce escape hatch. Every variant carries a reference to the widget it concerns; on the free-form announcement the reference is optional — present when a widget emitted it, null for app-level announcements. The rendering vocabulary for all of these lives in `SpeechRenderer`, never in the engine: the engine emits structure, the renderer emits words. Editor events deliberately carry only perceptual content (the grapheme, the echoed word, the spoken context) — never document text; a reader that needs more queries the widget. A read-only editor swallows typing silently — nothing was inserted, so there is nothing to echo — and lets Enter fall through to the layer's primary widget. Program notifications (a button's activation, a widget's `Changed`) ride the same queue as deferred callbacks and are delivered to the widget's own events at drain time — handlers never run inside input dispatch, so they may freely open dialogs or remove widgets. Tick events report elapsed ticker intervals. Programs react to widget events; readers react to accessibility events.

## 7.2. Draining and coalescing

The app drains the event queue when it chooses (`DispatchEvents`; the event loop drains every iteration, headless hosts after each input or mutation batch). Coalescing is applied at drain time: for state-describing events (focus, selection, item/tab/slider/toggle/filter changes, label and state deltas), only the latest instance of each event kind survives a drain — an intermediate focus move is discarded in favor of the settled one, with the label deltas keyed per property so a rename and a description change both survive; transient events (typing echoes, announcements) and all program notifications survive in order. The settled focus event is additionally delivered last in its batch: focus describes where the user is now, everything else describes what just happened, so a handler that closes a dialog and then announces its result is heard as result first, restored focus second, regardless of emission order. Interruption policy — silencing speech on a keypress, utterance length limits — belongs to the speech reader, not the engine: the app broadcasts `OnInterrupt` to every reader when a key goes down (hosts driving a headless app call `SruiApp.Interrupt` themselves). There is deliberately no urgent announcement variant: every keypress already silences speech, and cutting speech off from a timer or background event is bad screen-reader manners — asynchronous urgency belongs to earcons, not interruption.

## 7.3. Readers

A reader is any consumer of accessibility events: implement `IReader.OnEvent` and attach with `SruiApp.AddReader`; every attached reader hears every event, in order. The reference reader is the self-voicing `SpeechReader`: it renders events to utterances with the `SpeechRenderer` library and forwards them to Prism, and a windowed app installs one automatically. The rendering library composes announcements from the golden six in NVDA order and handles character echo (punctuation expansion, capital indication). Anticipated additional readers include braille, a UIA provider (the retained tree maps directly onto platform accessibility trees), and logging readers — the test suite's recording reader is one.

# 8. Focus and Navigation

Enter and Escape follow the Windows dialog convention: each layer may designate a primary and a cancel widget, Enter activates the primary and Escape the cancel, and only buttons claim Enter for themselves. A primary or cancel widget that cannot currently act — disabled, hidden, or inside a hidden subtree — does not activate; the input falls through unconsumed. Non-button widgets never claim Enter — there is no per-widget "item activated" event; acting on a list selection is the primary widget's job (it reads the selection) or that of an explicitly bound command. Activation is a `Widget`-level concept: any widget targeted by primary/cancel routing or an activate shortcut raises its `Activated` event.

Navigation has three axes. The tab ring visits every focusable node in depth-first tree order with wraparound; hidden subtrees are skipped, and groups and labels are never focusable. Disabled widgets stay in the ring: in a keyboard-only, screen-reader-first UI, skipping them would make them undiscoverable, so they are announced ("unavailable") but inert — input, key bindings, shortcuts, and activation all pass them by. Only hiding removes a widget from navigation. Hierarchy navigation moves along the tree structure itself: to parent, first visible child, or previous/next sibling with wraparound. Widget-internal navigation (arrows inside a list or edit box) is the focused widget's own affair and takes precedence through the claim order.

Any widget may carry shortcuts: key combos with a per-shortcut action — jump to the widget, activate it (its `Activated` event fires, without a focus move), or both, jump first. A widget may carry any number of shortcuts, and the first one added is the one focus announcements speak (the shortcut slot of the golden six). Shortcuts fire only while their widget is reachable — visible, enabled, outside hidden subtrees — and when several widgets bind the same combo, the first reachable claimant in depth-first tree order wins. Mnemonics (Alt plus a letter) are the conventional jump-action case; the framework treats them as ordinary shortcuts on the Alt+letter combo.

Focus memory remembers the last-focused child of each container so re-entering a container restores position. When the focused node is removed, focus recovers to the nearest surviving focusable node; when a layer is popped, the previous layer's focus is restored. Focus changes emit a focused accessibility event exactly once per settled change.

# 9. Commands and Key Bindings

Key bindings are host policy. The engine stores no command maps and dispatches no commands: input the claim order leaves unconsumed reaches the app's `UnhandledInput` hook, which matches it against whatever binding scheme the application likes — as do command palettes, categories, jump lists, and persistence.

The toolkit contributes three mechanisms to that host machinery, all public. `KeyCombo.ReservedReason` names the combos the framework itself consumes — plain Tab and Shift+Tab (the focus ring), Alt+letter (mnemonics), Alt+arrows (hierarchy navigation) — each with a spoken refusal for bind dialogs to announce; everything else, including Ctrl+Tab and Escape, is categorically bindable. Bind dialogs query `SruiApp.ReservedReasonFor(combo)` rather than the static property: it adds whatever the app's host reserved (`SruiApp.HostReservations` — under a multi-app host, the app-switching combos; section 10), so the same dialog refuses correctly standalone and hosted. `Widget.ReservesKey` reports which keys a widget consumes during normal interaction, so a bind dialog can warn about combos the focused widget would swallow — custom-authored widgets participate by overriding it. And the shortcut field captures any bindable combo before the framework can interpret it — including Alt+arrows and mnemonics — deliberately releasing only Tab and Escape so the keyboard user can always leave the field.

# 10. Multi-App Hosting

`MultiAppHost` runs several apps in one window: it owns the single `SdlHost`, one shared speech reader over Prism, and one shared sound manager, and multiplexes them across any number of hosted apps. Apps are created through `MultiAppHost.Add(name)` — at startup or at any point while running — which returns a `HostedApp`: a headless `SruiApp` (built into normally) pre-wired to the shared services — the system clipboard, the shared reader, the shared audio (`SruiApp.Audio` on a hosted app resolves to `MultiAppHost.Audio`, one device for the whole program, ticked and disposed by the host), and the host's key reservations. One windowed host per process — SDL's event queue is process-global.

An app leaves through `HostedApp.Close()`: it is removed from the host's list (switching skips it, messages to it drop silently), its `SruiApp` is disposed, and when it was the active app its neighbor is activated with the full switch announcement; closing a background app is silent. Close is idempotent and belongs at drain time, like any operation reaching beyond a widget's own state. A hosted app's own `Quit` is turned into `Close` by the host on the next tick, so an app's exit path behaves identically standalone and hosted. `HostedApp.IsClosed` and its `Closed` event report the transition, and `MultiAppHost.AppsChanged` fires on every add and close — the refresh signal for a launcher or task-list app reading `MultiAppHost.Apps`.

Exactly one app is active at a time; it receives all pumped input through its ordinary claim order, and the host's own `UnhandledInput`/`UnhandledKey` hooks fire when the active app declines. The rest keep running — clocks advance, tickers fire, messages deliver, state mutates — but silently. What a switch says is graded by how the user got there (`SwitchAnnouncement`, a parameter of `Activate`): the switching combos (`NextAppCombo`/`PreviousAppCombo`, default ctrl+tab and ctrl+shift+tab, host-configurable and nullable) name the destination — the app's name followed by its focused widget with context, the window-switch announcement — while a task-list pick passes `FocusOnly`, because the user just chose the name and repeating it is noise, and `Silent` says nothing at all (the caller speaks, or the switch is an invisible restore; the very first focus an app ever establishes still announces itself). `HostedApp.Name` is read at announcement time, so an app renaming itself — an editor adopting its document — speaks correctly from the next switch without ceremony. The switching combos are matched before the active app sees the input, so no app can trap the user, and they are hard-reserved inside every hosted app: `SruiApp.ReservedReasonFor(combo)` — the query bind dialogs use in place of the static `KeyCombo.ReservedReason` — answers with the framework's categorical reservations plus whatever the host claimed through `SruiApp.HostReservations`, evaluated live so a reconfigured switch combo moves the refusal with it. Deactivation runs the app's `FocusLost` (held-key state zeroes, exactly as when the window loses focus) and maintains `SruiApp.IsForeground`, a host-set flag app code branches on — the engine ignores it.

Background speech is policy, applied where the shared reader attaches: a background app's events are dropped, except announcements from apps that set `HostedApp.AnnouncesInBackground` — the flag passes `Announce` events only, never focus or state chatter, because asynchronous urgency from an app the user is not in belongs to announcements it chose to make (or to earcons). Readers attached directly to a hosted `SruiApp` bypass the policy and hear everything, which is the seam for per-app logging.

Apps message each other through the host: `HostedApp.Send(object)` queues into the target's mailbox, and its `MessageReceived` event delivers at drain time — outside input dispatch, on the one thread, whether or not the target is active. A message sent from a delivery handler (including to the sender's own app) waits for the next drain, so self-sends cannot loop a drain forever. Receivers pattern-match on the message type; the toolkit imposes no message vocabulary.

The host's loop mirrors the app's: `Run` is `Tick` in a loop (pump and route, tick shared audio, deliver messages, tick every app), `Quit` ends it, and a headless host (`MultiAppHost.Headless`) is driven through `HandleInput`/`HandleKey`/`DispatchEvents` — the tests' harness. `MultiAppDemo` exercises the whole surface: a Home app whose live task list (rebuilt on `AppsChanged`, activating with `FocusOnly`) fronts three others, combo switching, messaging, the background-announcement flag, and self-closing.

# 11. Native Boundaries

Native code sits behind three hand-written P/Invoke surfaces, each over a DLL staged by `native/build-native.ps1`: SDL3 (window, event queue, clipboard), Prism (speech/braille through the running screen reader or platform TTS), and cosmos with phonon beside it (game audio; consumed by `Srui.Audio`). Strings cross these boundaries as UTF-8; native booleans marshal as one byte; there are no callbacks on the UI path — the host polls SDL and drains the engine. Struct layouts and constants are transcribed from the vendored native headers, not from documentation.

The toolkit is not thread-safe and does not need to be: one `SruiApp` belongs to one thread. Multiple apps on different threads are fine.

The polling loop is allocation-free at idle. An empty pump or drain returns a shared empty batch rather than a fresh list, so an idle application's memory stays flat instead of accruing garbage at the loop cadence. Batches returned by `SdlHost.Pump` are therefore read-only by contract.

# 12. Testing

The toolkit runs fully headless; `Srui.Net.Tests` (xUnit) is the behavioral safety net. Behavioral tests drive the public surface end to end: build widgets in a headless `SruiApp`, push logical input, and assert what a recording reader hears as rendered utterances. Engine invariants are property-tested with seeded-random generators, including: the tab ring lands only on valid focusable nodes and visits each exactly once per cycle; word-boundary navigation always makes progress and stays in range; the editor's cursor and selection stay within the document under arbitrary operation sequences; the rope agrees with a reference implementation under thousands of random edits. New behavior is expected to arrive with its invariants encoded the same way.
